# v0.29.0-rc.1 — Real Portable OAuth Spike

## Purpose

This release adds a controlled OAuth 2.0 Authorization Code + PKCE spike for the portable account provider mode.

The feature is intentionally not production OAuth. It is a dev spike that proves the architecture can:

- generate a real PKCE `code_verifier` and S256 `code_challenge`
- build an authorization URL
- parse a redirect callback
- exchange an authorization code through the backend
- sanitize token responses into hashes/status metadata
- reject refresh until a server-side token vault exists
- export no raw access token, refresh token, authorization code, or PKCE verifier

## Standards basis

PKCE uses the S256 transform:

```text
code_challenge = BASE64URL-ENCODE(SHA256(ASCII(code_verifier)))
```

Browser OAuth remains risky if tokens are exposed to frontend state. This spike therefore keeps the verifier in `sessionStorage` runtime state and returns only sanitized token hashes from the backend.

## Backend endpoints

```text
POST /api/oauth/token-exchange
POST /api/oauth/refresh
```

`/api/oauth/token-exchange` performs a dev authorization-code exchange against an allow-listed token endpoint.

`/api/oauth/refresh` intentionally returns:

```text
oauth_refresh_requires_token_vault
```

Refresh is explicit and blocked until a backend token vault is designed.

## Export safety

Exported packets include:

```json
{
  "portable_oauth_spike": {
    "raw_token_exported": false,
    "access_token_exported": false,
    "refresh_token_exported": false,
    "code_verifier_exported": false,
    "production_ready": false,
    "threat_model_required": true
  }
}
```

Forbidden in exports:

```text
access_token
refresh_token
id_token
raw_token
code_verifier
client_secret
authorization header
```

## Non-negotiable guardrails

- Manual/private mode remains first-class.
- Mock portable account remains available.
- No live provider inference is enabled through OAuth.
- OAuth token exchange returns sanitized metadata only.
- No token vault is implemented.
- Real production merge still requires a threat model.
