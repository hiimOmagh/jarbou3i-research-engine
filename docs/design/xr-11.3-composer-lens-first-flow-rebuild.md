# XR-11.3 — Composer Lens + First-Flow Rebuild

## Status

Implementation package. RC-1 remains blocked until the product structure and visual evidence pass manual audit again.

## Context

XR-11.2 froze a full Product Structure Audit Freeze. The audit rejected narrow screenshot patching and identified the composer as the highest-leverage rebuild target. The visible blocker was not only decorative quality; it was product structure. The first screen still failed to communicate the primary path cleanly.

The main product path must read as:

```text
Topic → Lens → Copy Prompt
```

Before this patch, the Analysis Lens selector looked like an empty control beside a selected Biopolitical card. That empty-pill illusion made the primary composer look broken even though the browser and export contracts were green.

## Goals

1. Rebuild the Analysis Lens selector as two explicit choices: Strategic and Biopolitical.
2. Remove the empty-pill illusion from the first-screen composer.
3. Reduce first-screen vertical height without hiding required browser-tested controls.
4. Reframe the composer around Topic → Lens → Copy Prompt.
5. Compress repeated first-step guidance instead of adding more decoration.
6. Keep Simple Mode guided and Expert Mode powerful.
7. Preserve runtime IDs, exports, RTL/mobile behavior, and the XR-8 evidence matrix.

## Non-goals

- No provider changes.
- No backend changes.
- No schema changes.
- No OAuth/storage/database changes.
- No export contract change.
- No replacement of browser-tested IDs such as `#analysisLens`, `#topicInput`, `#copyPromptBtn`, `#jsonInput`, or `#exportHtml`.

## Structural changes

### 1. Analysis Lens selector

The lens selector remains the same DOM radiogroup:

```text
#analysisLens
button[data-lens="strategic"]
button[data-lens="biopolitical"]
```

The change is visual and structural: the selector now behaves like two explicit product choices instead of a segmented control that can appear empty when one option is active.

The active state now receives a specific high-specificity background and foreground treatment, preventing the active label from becoming white-on-white or visually missing.

### 2. First-flow hierarchy

The composer is tightened around the actual product sequence:

```text
1. Topic input
2. Analysis language and lens selection
3. Copy prompt action
4. Import/review/export as later workflow stages
```

The layout reduces excessive margins around the task header, topic field, lens field, actions, mini guide, and workflow guidance.

### 3. Mission and guidance compression

The mission card and workflow guidance remain present to preserve tested accessibility and guidance surfaces, but their visual weight is reduced. Repeated onboarding copy is treated as support text instead of competing with the primary composer.

### 4. Mobile and RTL behavior

Mobile keeps the lens cards readable and prevents overflow. On narrow screens the two lens choices can stack, while Arabic RTL padding and pseudo indicators remain mirrored.
