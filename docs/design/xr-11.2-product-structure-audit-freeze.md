# XR-11.2 — Product Structure Audit Freeze

## Decision

XR-11 and XR-11.1 improved the visual direction and repaired the mobile hero stack, but the manual review exposed a deeper problem: screenshots were being used as isolated visual checks instead of as evidence for the full product structure.

XR-11.2 freezes the product-structure audit before any further visual patch. The goal is to inspect the existing features, their placement, their value, and their relationship to the user journey. The next UI change must be derived from this audit, not from a single screenshot defect.

## Current release state

The codebase is technically stable enough for deeper product review:

- Browser core contract is green.
- Hosted evidence capture is green.
- Stable release readiness is green.
- No-browser and workspace hygiene are green.
- XR-8 evidence matrix exists and remains the visual audit substrate.

However, release-candidate tagging remains blocked until the product structure is coherent enough to support a public demo without explaining around confusing surfaces.

## Product intent

The product should feel like an intelligence workbench, not a form collection.

The user journey should be:

1. Choose the analytical frame.
2. Define the topic.
3. Generate/copy a high-quality prompt.
4. Import the AI structured answer.
5. Inspect review layers and diagnostics.
6. Export a credible report.

Everything visible on screen should support that path or be clearly deferred to expert use.

## Audit method

Every feature is classified into one of five actions:

| Action | Meaning |
|---|---|
| KEEP | Works structurally and visually; preserve as-is except minor polish. |
| POLISH | Useful and mostly correct, but needs visual/detail refinement. |
| REBUILD | Valid feature idea, but current implementation is structurally wrong. |
| HIDE | Useful only in secondary/expert context; should not burden the primary path. |
| REMOVE | Duplicate, confusing, or low-value surface that weakens the product. |

## Screen-by-screen structure audit

### 1. First screen / composer

Current structural issue: the first screen contains too many adjacent concepts: brand, mode selection, hero, AI motif, topic, language, lens, advanced controls, prompt actions, status, how-to guidance, and step navigation.

Required direction:

- The first screen must become a compact command-start surface.
- Topic + lens + copy prompt should be the visible spine.
- Advanced/options/examples should be secondary.
- Repeated instructions should be consolidated.

Priority: high.

### 2. Simple Mode journey

Current structural issue: Simple Mode is visually improved but still exposes expert concepts and duplicated guidance.

Required direction:

- Simple Mode should hide advanced options by default.
- Simple Mode should use one workflow spine, not stepper + guide + how-to + hero repetition.
- The import station should appear sooner after prompt generation.

Priority: high.

### 3. Expert Mode journey

Current structural issue: Expert Mode correctly exposes more controls, but it still reads partly as a long form instead of a command cockpit.

Required direction:

- Expert Mode can keep advanced options visible.
- Expert Mode should group controls by task: prompt configuration, import validation, review diagnostics, export readiness.
- Empty dashboards should explain what unlocks them without consuming too much height.

Priority: medium-high.

### 4. Mobile / RTL journey

Current structural issue: XR-11.1 fixed the hero stack, but mobile still has density pressure. Arabic/RTL can be technically correct while still feeling oversized or over-framed.

Required direction:

- Keep mobile header compact.
- Keep hero stacked and centered.
- Avoid side-by-side controls that create empty pills or clipped surfaces.
- Use RTL-native spacing and readable typography, not just mirrored LTR layout.

Priority: high.

### 5. Review and export states

Current structural issue: review/export sections are closer to product-grade than the first screen. The main risk is empty-state heaviness and unclear activation before import.

Required direction:

- Keep review/export architecture.
- Make inactive states compact and useful.
- Do not rebuild review/export unless concrete evidence shows a blocker.

Priority: medium.

## Next implementation milestone

The next implementation milestone should be:

**XR-11.3 — Composer Lens + First-Flow Rebuild**

Scope:

1. Rebuild the Analysis Lens selector into two explicit lens cards/pills.
2. Remove the empty lens-control illusion.
3. Consolidate repeated start/topic copy.
4. Reframe first-screen hierarchy around topic, lens, and Copy prompt.
5. Keep Simple/Expert rules intact.
6. Preserve all IDs, export contracts, RTL/mobile behavior, and XR-8 evidence matrix.

Do not change backend/provider/schema/OAuth/storage behavior.

## Closure criteria after XR-11.3

The product can return to RC-1 only if:

- `desktop-first-screen.png` has no broken/empty-looking controls.
- `mobile-first-screen.png` remains stacked, compact, and readable.
- Simple Mode reads as a guided workflow, not a long form.
- Expert Mode reads as a power-user cockpit, not a dense settings page.
- Visual evidence matrix passes.
- Manual visual audit classifies only non-blocking polish items.

## Frozen decision

XR-11.2 freezes the full-structure audit and blocks further release-candidate work until the feature hierarchy issues above are addressed.

RC-1 remains blocked.
XR-11.3 should be the next implementation patch.
