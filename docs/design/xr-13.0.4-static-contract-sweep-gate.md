# XR-13.0.4 — Static Contract Sweep Gate

## Purpose

XR-13 exposed a process failure: static checks were validating literal HTML attribute order instead of user-facing behavior and stable DOM semantics. That caused repeated token hotfixes even after browser evidence passed.

This hotfix changes the gate from brittle substring-only matching to semantic attribute-contract matching for index.html contracts while keeping the existing runtime, browser, export, RTL, schema, and evidence boundaries unchanged.

## What changed

- Added an attribute-order-insensitive contract helper to `tests/static-check.mjs`.
- Preserved existing contract tokens and failure messages.
- Kept browser-visible system anchors intact.
- Reordered the interface mode banner attributes as a compatibility belt-and-braces fix.
- No runtime behavior change.
- No CSS behavior change.
- No schema change.
- No import/export behavior change.

## Why this exists

The previous XR-13 hotfix sequence repaired one static token at a time:

1. `welcomeCard` visibility
2. `workflowPanel` clarity token
3. `workflowGuidance` token
4. next pending blocker: `interfaceModeBanner` token

That pattern is not acceptable for a user-friendly UX recomposition workflow. The correct fix is to sweep the contract mechanism so equivalent DOM semantics do not fail because attributes appear in a different order.

## Gate rule from this point

Before delivering another UI package, perform a static contract sweep:

1. Identify browser-required DOM anchors.
2. Identify static-check index.html attribute contracts.
3. Validate attributes semantically, not by fragile raw ordering.
4. Preserve visual simplification only when system anchors remain visible and accessible.
5. Avoid deleting, hiding, or renaming browser-tested controls unless tests are intentionally updated in the same package.
