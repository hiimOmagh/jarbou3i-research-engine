# v0.29.0-rc.1 — Real Backend Provider Hardening

This release hardens the optional hosted backend proxy. It does not add real OAuth, real source fetching, cloud accounts, or mandatory backend dependency.

## Scope

The Cloudflare Worker proxy now has explicit production-oriented controls:

- Structured error taxonomy: `error_code`, `error_category`, `retryable`, `request_id`.
- CORS allow-list rejection with `cors_origin_not_allowed`.
- Configurable rate limiting via `RATE_LIMIT_SECONDS`.
- Request body and prompt limits via `MAX_BODY_BYTES` and `MAX_PROMPT_CHARS`.
- Upstream response limit via `MAX_UPSTREAM_BYTES`.
- Provider timeout via `PROVIDER_TIMEOUT_MS`.
- Optional model allow-list via `ALLOWED_MODELS`.
- Redacted metadata-only audit logging via `AUDIT_LOGS_ENABLED`.
- Health endpoint exposes hardening status without secrets.

## Release gate

Backend mode is still optional. Before public backend use, run:

```bash
npm run test:backend
npm run test:v025:no-browser
npm run test:browser:provider
```

## Non-negotiables

- Browser-supplied API keys are stripped before upstream calls.
- Worker responses must never contain `OPENAI_API_KEY` or client-side BYOK values.
- Audit logs must not include prompts, provider responses, authorization headers, or raw secrets.
- Source task endpoint remains planning-only and performs no live source fetching.
- Manual/private mode must work without the backend.

## Environment variables

```text
OPENAI_API_KEY             required secret
OPENAI_BASE_URL            optional upstream endpoint
OPENAI_MODEL               default model
ALLOWED_MODELS             comma-separated server allow-list
ALLOWED_ORIGINS            comma-separated browser origin allow-list
MAX_PROMPT_CHARS           prompt character limit
MAX_BODY_BYTES             request body limit
MAX_UPSTREAM_BYTES         upstream response limit
PROVIDER_TIMEOUT_MS        upstream fetch timeout
RATE_LIMIT_SECONDS         per-client task cooldown
AUDIT_LOGS_ENABLED         true enables metadata-only audit logs
```
